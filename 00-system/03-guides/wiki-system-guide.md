# Wiki 시스템 가이드 — 지식이 복리로 쌓이는 곳

> **준비물**: Claude Code 설치 완료, do-better-workspace 클론 완료
> **대상**: Wiki를 처음 접하는 분. 이 문서 하나로 "이게 뭔지 / 왜 쓰는지 / 어떻게 쓰는지"를 잡습니다.

---

## 한 줄 정의

`30-knowledge/00-wiki/`는 **당신이 다루는 주제별 지식이 한곳에 모이고, 새 자료가 들어올 때마다 더 정교해지는 위키**입니다.

회의록·영상·기사·대화에서 나온 내용이 흩어져 사라지지 않고, 주제 페이지 하나에 계속 쌓입니다. 쌓일수록 가치가 올라갑니다. 이걸 **복리(compounding)**라고 부릅니다.

---

## 왜 필요한가

AI와 일하다 보면 같은 설명을 반복하게 됩니다.

- 어제 정리한 내용을 오늘 또 처음부터 설명
- 한 달 전 회의에서 나온 결론이 어디 갔는지 못 찾음
- 비슷한 주제를 매번 새 파일로 만들어서 파편화

Wiki는 이걸 막습니다. **주제에 대해 물으면 Claude가 위키부터 확인**하고, 새로 알게 된 건 **다시 위키에 통합**합니다. 당신의 지식이 AI가 매번 다시 읽을 수 있는 형태로 남습니다.

> 핵심 철학: "지식은 한 번 정리되고 계속 최신으로 유지된다. 질문할 때마다 처음부터 다시 만들지 않는다." (Karpathy의 LLM Wiki 아이디어)

## 어디에 있고 무엇으로 되어 있나

```
30-knowledge/00-wiki/
├── SCHEMA.md      # 운영 규칙 (페이지 형식, 3오퍼레이션) — AI가 참고하는 설계도
├── index.md       # 전체 토픽 카탈로그 — "어떤 주제가 있나" 찾을 때 여기부터
├── log.md         # 활동 기록 (언제 무엇을 넣었나, append-only)
├── README.md      # 짧은 운영 가이드
└── [토픽].md       # 실제 토픽 페이지들 (예: 동사-중심-사고.md)
```

처음 클론하면 토픽 페이지는 거의 비어 있습니다. **당신이 자료를 넣으면서 채워가는 구조**입니다.

### 3계층 — 누가 무엇을 관리하나

| 계층 | 위치 | 역할 |
|------|------|------|
| **원본 소스** | `10-projects/`, `41-daily/`, `00-inbox/` 등 | 회의록·노트 원본. **수정 안 함, 읽기만** |
| **위키** | `00-wiki/` | 소스에서 뽑은 지식이 통합되어 쌓이는 곳. AI가 쓰고, 당신이 읽고 승인 |
| **스키마** | `SCHEMA.md` + 위키 스킬 | 운영 규칙. 쓰면서 같이 다듬어짐 |

## 어떻게 쓰나 — 3가지 동작

Wiki는 슬래시 커맨드를 외울 필요 없습니다. **자연어로 말하면** Claude가 알아서 합니다.

### 1. Ingest (넣기) — 자료를 위키에 통합

자료가 생겼을 때 위키에 쌓습니다.

```
이 회의록 위키에 반영해줘
방금 본 이 영상 내용 위키에 넣어줘
이번 세션 내용 지식 축적해줘
```

→ `wiki-ingest` 스킬이 실행되어:
1. 소스를 읽고 핵심을 뽑고
2. 기존 토픽 페이지 중 관련된 걸 찾고
3. 거기에 근거를 추가하고, 핵심 요약을 갱신하고, 다른 페이지와 연결합니다.

비슷한 토픽이 이미 있으면 **새 파일을 만들지 않고 기존 페이지를 확장**합니다. 이게 파편화를 막는 핵심입니다.

### 2. Query (꺼내기) — 위키에서 찾아 답하기

주제에 대해 물으면 Claude가 `index.md` → 검색 → 해당 페이지 순으로 확인하고 답합니다.

```
동사 중심 사고에 대해 정리된 거 보여줘
AX 컨설팅 관련해서 우리가 쌓아둔 내용이 뭐야?
```

→ 대화에서 나온 분석이 가치 있으면 **그 결과도 다시 위키에 저장**됩니다. 묻는 행위 자체가 위키를 키웁니다.

### 3. Lint (점검) — 위키 건강검진

가끔 위키 상태를 점검합니다.

```
위키 점검해줘
wiki-lint
```

→ `wiki-lint` 스킬이 고아 페이지, 죽은 링크, 오래된 페이지(90일+), 모순, 누락된 연결을 찾아 정리를 제안합니다.

## 이건 위키에? daily에? — 어디 넣을지 헷갈릴 때

처음엔 "이걸 위키에 넣어야 하나, 그냥 노트에 적나"가 자주 막힙니다. 기준은 **"나중에 다시 꺼내 쓸 지식인가"**입니다.

| 이런 내용은 | 여기로 | 왜 |
|------------|--------|-----|
| 오늘 한 일, 일정, 그날의 메모 | `41-daily/` (daily note) | 시간순 기록. 그날 지나면 거의 안 봄 |
| 빠르게 떠오른 생각, 아직 정리 안 됨 | `00-inbox/` | 임시 캡처. 나중에 분류 |
| **반복해서 쓸 개념·노하우·결론** | **`00-wiki/`** | **주제별로 쌓이고 다시 꺼내 쓰는 지식** |
| 특정 프로젝트 진행 자료 | `10-projects/` | 그 프로젝트가 끝나면 archive |

핵심 구분: **"이 주제 또 나올 것 같다"** 싶으면 위키. **"오늘만 의미 있다"** 싶으면 daily.

판단이 애매하면 일단 daily나 inbox에 적어두고, 나중에 "이 내용 위키에 반영해줘"로 옮기면 됩니다. 원본은 그대로 두고 지식만 위키로 올라갑니다.

## 토픽 페이지는 어떻게 생겼나

각 페이지는 정해진 형식을 따릅니다.

```markdown
# [토픽명]
> **관련**: [[토픽A]] · [[토픽B]]      ← 다른 페이지와의 연결
> **Facets**: 키워드1 · 키워드2          ← 검색용 핵심어

## 핵심        ← 지금까지 아는 것의 요약 (가장 먼저 읽는 부분)
## 근거        ← 소스별 증거. 모든 항목에 [source: 출처, 날짜] 인용
## 적용        ← 내 맥락에서 실제로 어떻게 쓰는가
## Open Questions  ← 아직 모르는 것
## Related     ← 관련 토픽 링크
```

**핵심**만 읽으면 그 주제의 현재 결론을 알 수 있고, **근거**를 보면 어디서 나온 얘기인지 추적할 수 있습니다.

## 실제로 어떻게 채워지나 — 넣기 전과 후

클론 직후 위키는 비어 있습니다. 다음은 자료 한 줄을 넣었을 때 무슨 일이 일어나는지 보여주는 예시입니다.

**① 넣기 전 — 원본 (예: 어제 회의록 한 줄)**

> "고객은 우리가 얼마나 전문적인지보다, 매장에 들어왔을 때 느끼는 분위기로 판단한다는 얘기가 나왔다."

**② 당신이 한 말**

```
이 회의록 위키에 반영해줘
```

**③ 넣은 후 — Claude가 만든 `감각.md`**

```markdown
# 감각
> **관련**: [[소비자-중심]]
> **Facets**: 고객 경험 · 분위기

## 핵심
고객은 전문성이 아니라 오감의 총체로 경험을 판단한다.

## 근거
- 매장의 분위기가 전문성보다 판단 기준이 된다 [source: 0617-회의록, 2026-06-17]

## Open Questions
- 분위기를 구체적으로 어떤 요소(조명·소리·향)가 만드나?
```

회의록 한 줄이 **출처가 붙은 토픽 페이지**가 됐습니다. 다음에 비슷한 얘기가 또 나오면, 새 파일을 만들지 않고 **이 `감각.md`의 근거에 한 줄이 더 쌓입니다.** 근거가 쌓일수록 `## 핵심` 요약이 더 정확해집니다. 이게 다음에 설명할 복리입니다.

---

## "복리"가 정확히 무슨 뜻인가

단순히 파일 개수가 느는 게 아닙니다. **통합(integration)**이 일어나야 복리입니다.

새 자료 하나가 들어오면:

1. **근거가 두꺼워짐** — 기존 페이지에 새 증거가 쌓임
2. **핵심이 정교해짐** — 새 근거가 기존 이해를 바꾸면 요약이 업데이트됨
3. **연결이 강해짐** — 한 자료에서 여러 토픽이 동시에 풍부해지고 서로 링크됨
4. **모순이 잡힘** — 기존 주장과 충돌하면 `[!contradiction]` 표시 (삭제하지 않고 판단은 당신이)
5. **질문이 풀림** — Open Questions에 있던 게 새 자료로 답을 얻음

같은 자료라도 그냥 폴더에 던져두면 죽은 파일이지만, 위키에 통합하면 **여러 페이지를 동시에 키우는 자료**가 됩니다.

---

## 실제로 이렇게 자랍니다

이 시스템은 이론이 아니라 실제 운영 중인 구조입니다. 이림의 PKM 위키는 같은 4개 파일에서 시작해 현재:

- **50개 이상의 토픽 페이지** (동사 중심 사고, AX 컨설팅 운영, 비개발자 AI 교육 등)
- 진입을 돕는 **허브 토픽**과 카테고리별 카탈로그
- 어떤 페이지는 **근거 50개 이상**이 쌓여 별도 페이지로 분리될 만큼 두꺼워짐
- 수백 건의 ingest 기록이 `log.md`에 시간순으로 남음

처음엔 비어 있던 게 매주 자료를 넣으면서 이렇게 됩니다. **당신의 위키도 똑같이 시작해서 똑같이 자랍니다.**

## 처음 시작하는 사람을 위한 첫 3걸음

1. **첫 자료 하나 넣어보기**
   최근 회의록이나 정리하고 싶은 노트를 하나 골라서:
   ```
   이 파일 위키에 반영해줘: [파일 경로]
   ```
   Claude가 어떤 토픽으로 만들지, 어떻게 통합할지 제안합니다. 한 소스씩 같이 보면서 진행됩니다 (supervised mode).

2. **index.md 열어보기**
   방금 만들어진 토픽이 카탈로그에 등록된 걸 확인합니다. 여기가 위키의 목차입니다.

3. **2~3개 더 넣은 뒤 물어보기**
   자료가 몇 개 쌓이면:
   ```
   [주제]에 대해 위키에 정리된 거 종합해줘
   ```
   흩어진 자료가 하나의 답으로 합쳐지는 걸 경험하게 됩니다. 이 순간이 복리의 시작입니다.

---

## 꼭 기억할 규칙 4가지

- **기존 페이지 확장 우선** — 비슷한 토픽이 있으면 새 파일을 만들지 말 것
- **출처 인용 필수** — 모든 근거에 `[source: 출처, 날짜]`
- **모순은 삭제 금지** — `[!contradiction]` 표시만, 판단은 당신
- **원본은 안 건드림** — 위키는 소스에서 뽑은 지식만. 원본 노트는 그대로 둠
