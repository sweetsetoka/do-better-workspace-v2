# inbox-triage — raw 인박스 → 업무 라우팅 디스패처

## 책임 경계 (이게 왜 분리되는가)

수집(`kakao-read`의 카톡 체크 등) ↔ 업무 체계(todo·일정·info 보관) 사이의 연결 고리.
수집 스킬은 `00-inbox/raw/`에 `processed:false`로 떨구기만 하고 분류를 안 한다. 이 스킬이 "이게 무슨 종류 → 어디로"를 판정·위임한다 — 출처가 여럿이어도 라우팅은 여기 한 곳.

**이 스킬이 직접 하는 것**: raw 읽기 · 종류 판정 · proposal 제시 · processed 마킹 · info 요약 보관.

**위임하는 것** (전담 스킬에게):
- 할 일 등록 → `todo`
- 회신대기 추적 → `todo` (Waiting 항목)
- 일정 → `todo`에 기한 포함 (이 워크스페이스엔 캘린더 CLI가 필수 아님)
- 고가치 정보 축적 → `wiki-ingest`
- 긴 다주제 통화·미팅 구조화 → `transcript-organizer`

분류를 새로 발명하지 않는다. 판정 표로 1차 종류만 가르고, 무거운 처리는 전담 스킬이 한다.

---

## Step 0: 스캔

```bash
grep -lZ "processed: false" 00-inbox/raw/*.md 2>/dev/null
```

- 인자로 특정 파일명을 받으면 그 1건만 대상.
- **이미 마커가 붙은 건 신규에서 제외**: `processed:false`라도 frontmatter에 `routed_to:` 또는 `triage_held:`가 있으면 "추적/보류 중"으로 분리. 같은 항목에 같은 proposal을 반복하지 않는다. (held는 별도로 묶어 "기존 보류 N건"으로만 표시.)
- `raw/` 안에 미디어 첨부 폴더(`_media/` 등)가 있으면 source 파일이 아니므로 제외.

대상 0건이면 "raw 인박스 비어 있음(또는 전부 처리됨)" 보고하고 종료.

---

## Step 1: 로드

각 대상 파일에서:
- `source` 먼저 본다 → 식별 필드를 고른다 (예: `kakao-check`이면 `kakao_date` · `kakao_chat_id` · `kakao_room` · `channel`). 다른 수집원이 추가되면 그 source의 식별 필드를 같은 방식으로 읽는다.
- 공통: `direction`(in/out), `collected`
- 본문: `## 본문` 코드블록 + (있으면) `## 업무 후보` / `## 첨부 텍스트 내용`
- 첨부 경로가 본문에 적혀 있고 별도 텍스트 파일(통화 전사 등)을 가리키면 **그 파일도 Read**.

> **카톡 출처 메모**: `kakao-check`는 카톡 체크 스킬이 *업무 신호가 있을 때만* 떨군 1:1 창이라 1차 노이즈 게이트(2-A)에 거의 안 걸리고 본문 게이트(2-B)로 떨어진다(calendar/task/follow-up). 즉 카톡 raw는 이미 "사람이 보낸 의미 있는 것"으로 1차 선별된 상태다.

## Step 2: 종류 판정

cascading. 1차 게이트(frontmatter/표면 신호)로 거르고, 안 걸리면 2차(본문 내용)로.

### 2-A. 1차 게이트 (본문 깊이 안 봄)

| 신호 | 종류 | 신뢰도 |
|------|------|--------|
| 카드 승인 알림 (`승인안내`/`[Web발신]…승인` + 금액) | **noise/버림** | 0.95 |
| OTP·인증번호 (`인증번호`/`인증코드`/`verification` + 4~6자리) | **noise** | 0.95 |
| 배송·광고·쿠폰 (`배송`·`쿠폰`·`할인`·`이벤트` 단독) | **noise** | 0.85 |
| 통화 녹음/전사 텍스트 첨부 | 통화전사 → **2-A-call 게이트** | — |
| URL 단독 (행동요구 없는 링크) | 링크/info → **2-B로** | — |

### 2-A-call. 통화전사 2-tier 게이트 (성격이 주, 줄 수는 보조)

통화전사가 감지되면 **성격**으로 먼저 가른다. 줄 수는 보조 신호일 뿐 — 짧은 통화도 화자 분할로 길어지므로 줄 수 단독 판정 금지.

| 통화 성격 | Tier | 처리 |
|-----------|------|------|
| **단일 안건** (문의·약속 잡기·결정 1건. 대다수) | **Tier 1** | 이 스킬 인라인 경량 정제 → **2-B-call** |
| **다안건 미팅·강의·인터뷰** (여러 주제·긴 논의·아카이브 가치) | **Tier 2** | `transcript-organizer` 권고 (HITL). 전체 구조화는 그 스킬이 |

- 애매하면(중간 길이·안건 1~2개) Tier 1 우선 — 사실 카드로 빠르게 뽑고, 더 깊은 정리가 필요하면 그때 Tier 2 권고.

### 2-B. 2차 판정 (본문 내용)

| 본문 신호 | 종류 | 목적지 | 신뢰도 |
|-----------|------|--------|--------|
| 사람이 보낸 질문/요청/회신요망 | **follow-up** | `todo` (Waiting) | 0.7+ |
| 날짜+시간+장소 약속 | **calendar** | `todo` (기한 포함) | 0.75+ |
| 명시적으로 해야 할 일 | **task** | `todo` | 0.7+ |
| 참고·뉴스·링크 (행동 불요) | **info** | 00-inbox 보관 / 가치 높으면 `wiki-ingest` | 0.6+ |
| 확신 안 섬 / 신호 동률 모호 | **미분류** | 격리(held) | <0.7 |

판정 시 신뢰도(0.0~1.0)를 항목마다 명시한다. **<0.7은 미분류 격리** (Step 5).
신뢰도 앵커: frontmatter 기계 신호 = 0.9+ / 본문 단일 명확 신호 = 0.7~0.8 / 모호·복합 = <0.7.

### 2-B-call. 통화 경량 정제 (Tier 1) — STT 오인식 보정 + 사실 카드

짧은 업무 통화(Tier 1)는 **여기서 인라인으로 정제**한다. STT는 의미 오인식이 남는다(필러 반복, 음성비서 오작동, 화자 미식별 등). 700줄 미화 전사를 만들지 않는다 — **정확한 사실 5필드**만 뽑는다.

**1) 노이즈 무시** — 필러 반복("네 네." 연속), 음성비서 오작동, 단순 인사·맞장구·통화 시작/종료 의례는 사실 추출에서 버린다.

**2) 고유명사 보정 (창작 금지).** 회사명·사람 이름은 통화에서 들린 표기를 쓰되, 확신 없으면 `(미확인 — 표기 변형 후보)`로 둔다. 워크스페이스에 알려진 이름 목록(예: 프로젝트 노트·연락처 파일)이 있으면 grep으로 끌어와 음운 유사 표기를 보정하고 근거를 남긴다. **근거 없는 이름 단정 금지.**

**3) 화자 식별.** 누가 누구인지 맥락으로 판정. 불확실하면 **"(추정)"** 표기.

**4) 출력 = 사실 카드 (5필드).** 미화·장문 금지:
```
=== 통화 사실 카드 (Tier 1) ===
상대(회사/이름): <표기> (불확실하면 "(미확인)")
핵심 내용: <1~2문장>
약속/일정: <일시·장소(있으면) / 없으면 "없음">
해야 할 일: <있으면 1줄씩 / 없으면 "없음">
보정 노트: <교정 항목 + 제외한 노이즈>
```

**5) 핸드오프.** 카드의 약속은 calendar(`todo` 기한 포함), 해야 할 일은 task(`todo`)로. proposal-first 게이트 유지 — 카드는 제안이고, 등록은 사용자 승인 후.

## Step 3: 목적지 매핑 · 위임 인터페이스

| 종류 | 목적지 | 넘기는 것 | 처리 |
|------|--------|-----------|------|
| follow-up | `Skill todo` (Waiting) | 회신할 대상·내용 1줄 + context(출처 raw 경로) | 위임 |
| task | `Skill todo` | 내용 1줄 + 우선순위 추정 + context(출처 raw 경로) | 위임 |
| calendar | `Skill todo` (기한 포함) | 제목·일시·장소를 todo 내용에 + 기한 설정 | 위임 |
| info | 00-inbox 보관 / `wiki-ingest` | 요약 1~2줄 + 원문 링크. 가치 높으면 wiki-ingest 권고 | 자동(요약·보관) |
| Tier 2 통화 | `transcript-organizer` 권고 | 전사 파일 경로 | 위임(HITL) |
| noise | discard | — (파일 보존 + processed 마킹) | 자동 |

### 다중 목적지 — 한 항목이 여러 갈래로 보일 때
한 통화에 약속 + 해야 할 일이 같이 있으면 각각 calendar(todo 기한)·task(todo)로 쪼갠다. 단 같은 사실을 중복 등록하지 않도록 주의(약속 자체가 곧 할 일이면 하나로).

## Step 4: Proposal 제시 (HITL 게이트)

판정 결과를 **항목별 테이블 + 근거 1~2줄(본문 인용)**로 출력한다. 즉흥 라우팅 금지 — 반드시 이 proposal을 거친다.

```
=== inbox-triage proposal (raw/ 미처리 N건) ===

| # | 파일 | 종류 | 신뢰도 | 목적지 | 다음 동작 |
|---|------|------|--------|--------|-----------|
| 1 | kakao-1234 (○○님 1:1) | calendar | 0.85 | todo(기한) | "6/10 15시 미팅" → todo에 기한 포함 등록 |
| 2 | kakao-5678 (△△ 문의) | follow-up | 0.75 | todo(Waiting) | 회신 대기 항목으로 등록 |
| 3 | kakao-9012 (뉴스 링크) | info(경계) | 0.65 | info/보관 | HITL — 보관 vs "링크확인 todo" 결정 필요 |

근거:
- #1: "다음 주 수요일 3시에 봬요" — 일시 합의 닫힘 → calendar.
- #2: "그 자료 언제쯤 가능할까요?" — 회신요망 → follow-up.
- #3: 뉴스 URL 단독, 행동요구 약함, 신뢰도 0.65 → 격리/확인.

행별로: OK / 수정(목적지 변경) / 격리(보류) / 버림 을 알려주세요.
```

HITL 경계:
- **자동(승인 즉시)**: info 보관 · noise 마킹 · todo 등록.
- **HITL 필수**: 마감/일정 확정 · 미분류 처리.
- **미분류(<0.7)**: proposal에 올리되 사용자가 지정하기 전엔 건드리지 않음.

---

## Step 5: 라우팅 실행

사용자 승인(행별)대로:

- **task / follow-up / calendar** → `Skill todo` 호출(내용·우선순위·기한·context). → 완료 후 `processed:true`.
- **info** → 요약 1~2줄 + 원문 링크를 적절 노트에 보관(가치 높으면 `wiki-ingest` 권고). → `processed:true`.
- **Tier 2 통화** → `transcript-organizer` 권고(HITL). 그 스킬이 전체 구조화. → 완료 후 `processed:true`.
- **noise** → 라우팅 없음. 마킹만(파일 보존).
- **미분류** → 건드리지 않고 `triage_held:<이유>`만.

---

## Step 6: processed 마킹 (상태머신)

raw 파일 frontmatter를 Edit으로 갱신.

```
processed:false  (수집 초기값)
 ├ noise/info/todo/calendar 완료   → processed:true + routed_to:<dest> + triaged:<YYYY-MM-DD>
 └ 미분류/보류                      → processed:false + triage_held:<reason>
```

재처리 방지 핵심: `processed:false`라도 `routed_to`/`triage_held` 마커가 있으면 Step 0에서 신규로 안 잡힌다.

마킹 예 (Edit으로 frontmatter에 추가):
```
processed: true
routed_to: todo
triaged: 2026-06-03
```

## 알려진 한계

- **고유명사 보정 한계**: 워크스페이스에 알려진 이름 목록이 없으면 STT 표기를 못 고침 → "(미확인)"로 두고 사람이 확정. 근거 없는 이름 단정 금지.
- **Tier 1/2 경계**: 성격(단일 안건 vs 다안건)이 주, 줄 수는 보조. 중간 케이스는 Tier 1 우선 후 필요시 Tier 2 권고.
- **신뢰도 주관성**: 표의 앵커 값으로 고정하되, 경계값은 HITL로 흡수.
- **캘린더 직접 등록 안 함**: 이 워크스페이스엔 캘린더 CLI가 필수 아님. 일정은 `todo`에 기한으로 남긴다. (Google Calendar 연동을 쓰면 todo에서 처리.)
